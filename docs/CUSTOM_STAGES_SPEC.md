# Custom Stages Spec

Custom stages are entirely separate from stage skins (DAS). Stage skins are alternate
textures for the 6 legal stages; custom stages are wholly new stages added to the game
via MEX's stage system.

---

## Data Model

Each custom stage requires:

| Field | Source | Notes |
|---|---|---|
| `.dat` file | User-provided | The stage geometry/collision/code file |
| Stage name | User-provided | Display name (e.g. "Green Hill Zone") |
| Location/subtitle | User-provided | Shown on banner (e.g. "Sonic Series") |
| Screenshot | User-provided | Preview image for the vault card |
| Icon | User-provided OR derived | 64x56 CI8 image for stage select screen. Can be auto-generated by scaling screenshot down. |
| Banner | Auto-generated | 224x56 I4 grayscale image. Generated via `StageBannerGenerator.cs` using name + location with the correct Melee fonts ("A-OTF Folk Pro" 40pt / "Palatino Linotype" 18pt). |
| OG stage | Auto-detected or user-selected | The vanilla stage this was cloned from. Determines function pointers, CollisionMaterials, MapDescPointer. See detection section below. |
| 3D preview model | User-selected | The model shown in the corner of stage select. Selected from existing vanilla models by changing a flag. |
| Music playlist | User-configured | Which songs play on this stage. |

### Metadata storage

New top-level key in `storage/metadata.json`:

```json
{
  "custom_stages": [
    {
      "id": "green-hill-zone",

```

```

  ├── {stage-id}.dat
  ├── {stage-id}_screenshot.png
  └── {stage-id}_icon.png
```

## OG Stage Detection

When a custom stage is installed, we need to determine which vanilla stage it was
cloned from. This sets the function pointers (OnStageLoad, OnStageInit, etc.),
CollisionMaterials, and MapDescPointer — without these, the game crashes.

### How it works

The OG stage is identified by a set of RAM function pointer addresses that correspond
to vanilla Melee stages compiled into the DOL. Each vanilla stage (0-70) has a unique
set of pointers. Custom stages reuse a vanilla stage's pointers.

**The function pointers are NOT stored inside the .dat file.** They are external MEX
configuration stored in `stage.json`/`stage.yml` package metadata, or in the MEX
project's `MxDt.dat`.

### Detection tiers

**Tier 1 — Package has metadata (most common):**
Stage packages exported from MexManager/MexTool contain `stage.json` or `stage.yml`
with all function pointer values. We read `onStageLoad` and match against the vanilla
lookup table to identify the OG stage automatically.

**Tier 2 — Raw `.dat` file with no metadata:**
Cannot auto-detect. Default to **Targets!Sheik (ID 60)** which is the most commonly
used base for custom stages, but show a dropdown so the user can select a different
base if needed. Some stages use other bases (e.g. Peach's targets for Daisy-related
stages).

### Vanilla stage lookup table

Extracted from the Melee DOL at `0x803DFEDC`. The full table of 71 vanilla stages
with their OnStageLoad, OnStageInit, OnStageGo, CollisionMaterials, and
MapDescPointer values is static and never changes. Key entries:

| ID | Name | OnStageLoad |
|---|---|---|
| 36 | Battlefield | `0x80219D54` |
| 37 | Final Destination | `0x8021A7C4` |
| 40 | Targets!Mario | `0x8021F8B4` |
| 41 | Targets!C.Falcon | `0x8021FCD4` |
| 55 | Targets!Peach | `0x80222928` |
| 56 | Targets!Pichu | `0x80222C0C` |
| 60 | Targets!Sheik | `0x802238D8` |
| 65 | Targets!Ganon | `0x80224754` |

The complete table should be stored as a static lookup in the backend. Full extraction
was done from `main.dol` via the function table at `0x803DFEDC`, reading each
`MEX_Stage` struct (0x34 bytes) at the pointer for each index.

### Function pointers per OG stage (Targets!Sheik example)

```
CollisionMaterials:    0x803BEED8
MapDescPointer:        0x803E94B8
OnStageInit:           0x80223868
OnStageLoad:           0x802238D8
OnStageGo:             0x802238DC
OnDemoInit:            0x80223864
OnUnused:              0x80223900
OnTouchLine:           0x80223B38
OnCheckShadowRender:   0x80223B40
```

### grFunction node

Some custom stage `.dat` files contain a `grFunction` root node with embedded PPC
code. This provides the stage's own function implementations (OnStageInit, etc.)
which override or supplement the vanilla stage code. Stages with grFunction still
need an OG stage base set for CollisionMaterials/MapDescPointer. The grFunction
code is not parsed by MexManager — it's handled by the game's MEX runtime.

## UI

### Tab structure

The vault tabs change to:

```
Character Costumes | Stage Skins | Custom Stages (NEW) | Patches
```

- "Characters" renamed to "Character Costumes"
- "Stages" renamed to "Stage Skins"
- "Custom Stages" is new
- "Patches" unchanged

### Custom Stages list view

No subcategories (unlike character costumes which group by character, or stage skins
which group by the 6 DAS stages). Just a flat grid of stage cards.

Each card shows:
- Screenshot thumbnail
- Stage name

Same card styling as stage skin variant cards.

### Edit modal

Same modal pattern as stage skin edit modal, but with different fields:

**Left column:**
- Screenshot image (uploadable/replaceable)

**Right column:**
- Stage name (text input)
- Location/subtitle (text input)
- OG stage (dropdown of vanilla stages, auto-detected or user-selected)
- Icon preview (small 64x56 preview, uploadable or auto-derived from screenshot)
- 3D preview model selection (dropdown of existing vanilla models)
- NO slippi safety options (not applicable to custom stages)

**Bottom action bar:**
- Save / Export / Delete (same pattern as existing)

## Import Flow

### From zip package (with stage.json/stage.yml)

1. User drops zip file
2. Detect `stage.json` or `stage.yml` in the zip
3. Parse metadata: name, location, function pointers
4. Extract `.dat` file
5. Extract screenshot/icon if present
6. Auto-detect OG stage by matching `onStageLoad` against vanilla lookup table
7. Auto-generate banner via `StageBannerGenerator` from name + location
8. If no icon but screenshot exists, scale screenshot to 64x56 for icon
9. Save to `storage/custom_stages/`, update `metadata.json`
10. Show in UI for review/editing

### From raw .dat file

1. User drops `.dat` file
2. Prompt for: stage name, location/subtitle
3. Prompt for screenshot (optional)
4. Default OG stage to Targets!Sheik (ID 60), user can change via dropdown
5. Auto-generate banner from name + location
6. If screenshot provided, derive icon from it (scale to 64x56)
7. Save to `storage/custom_stages/`, update `metadata.json`

## Installation (to MEX project)

Installing a custom stage into the MEX project requires new MexCLI capabilities.
These don't exist today — MexCLI currently has zero stage commands.

### Installation steps

1. **Duplicate the OG stage entry** in the MEX project (e.g. clone Targets!Sheik)
2. **Set the new stage's name** and file path to the custom `.dat` file
3. **Copy the `.dat` file** into the MEX project's `files/` directory
4. **Set icon and banner assets** on the new stage entry
5. **Update stage select menu (MnSlMap):**
   - Create a new page if no empty slots exist
   - Place the new stage's icon on the page
6. **Set the 3D preview model** flag if user selected one

### MexCLI commands needed

Option A — Single consolidated command:
```
mexcli add-custom-stage <project> <dat_path> --name <name> --location <loc>
    --og-stage <id> --icon <path> --banner <path> [--preview-model <id>]
```

Option B — Granular commands:
```
mexcli list-stages <project>
mexcli duplicate-stage <project> <source_stage_id>
mexcli set-stage-file <project> <stage_id> <dat_path>
mexcli set-stage-name <project> <stage_id> <name> <location>
mexcli set-stage-assets <project> <stage_id> --icon <path> --banner <path>
mexcli add-stage-to-menu <project> <stage_id>
mexcli remove-custom-stage <project> <stage_id>
```

TBD: which approach. Granular gives more flexibility and is consistent with
how costume commands work.

## Icon Generation

Stage select icons are 64x56 pixels, CI8 format with RGB5A3 palette.

When the user doesn't provide a custom icon:
1. Take the screenshot image
2. Scale to 64x56 using bicubic resampling
3. Convert to CI8/RGB5A3 format via `ImageConverter` in mexLib

The game renders its own border around icons via `IconDoubleModel` in the stage
select screen — no border compositing needed in the icon texture itself.

User can optionally upload a custom icon image that overrides the auto-generated one.

---

## Banner Generation

Stage banners are 224x56 pixels, I4 (grayscale) format.

Generated automatically via `StageBannerGenerator.cs` in MexManager:
- Font 1 (stage name): "A-OTF Folk Pro" 40pt, skewed -10 degrees
- Font 2 (location): "Palatino Linotype" 18pt
- White text on transparent background
- Horizontal stretch up to 208px (name) and 160px (location)

This runs as part of the MexCLI stage installation, not at import time in the vault.
At vault import time we just store the name/location text and generate the banner
when actually installing to the MEX project.
