# Melee File Map

This page is a high-level map of the parts of Melee that Nucleus and the current research touch most often.

## Character Files

Character costume and model work usually centers on `Pl*.dat` files. These archives are where costume-specific assets, model data, and other character-side content tend to live.

In practice, a lot of modding work starts by identifying which costume archive owns the thing you want to change, then diffing that archive before and after a known-good mod.

Ice Climbers are the main special case:

- Popo and Nana use separate costume DATs
- one practical Ice Climbers costume pair usually spans both DATs

For the paired portrait and UI side of that rule, see [Ice Climbers Pairing](Ice-Climbers-Pairing.md).

## Effect Files

Effect work is split between character-specific `Ef*.dat` files and shared effect data such as `EfFxData.dat`.

That split matters because some visuals are owned by a character, while others are reused or shared globally. If an effect edit appears to affect multiple characters or multiple costumes, it is often a sign that the data is not costume-local.

## Stage Files

Stage modding usually works on stage archives rather than costume archives. Nucleus currently handles supported stage mods through the DAS workflow described in [Dynamic Alternate Stages](Dynamic-Alternate-Stages.md).

For now, the important distinction is that stage imports are a separate asset family with their own naming, validation, and Slippi-compatibility concerns.

## CSPs And UI Assets

CSP generation, alt CSP management, and pose capture are important parts of the current workflow, but they sit closer to presentation and tooling than to the core gameplay assets.

That means CSP workflows are relevant to mod packaging and polish even when the costume or stage data itself has not changed.

Ice Climbers are a special case here too. See [Ice Climbers Pairing](Ice-Climbers-Pairing.md).

## Why This Matters

Most Melee modding tasks boil down to two questions:

1. which archive actually owns the thing I want to change
2. is the data local to one asset, shared across assets, or generated by tooling

If those two questions are answered early, the rest of the workflow gets much simpler.
